# 06 · 反馈到钉钉 · 全套设置流程

> 站内右下角「反馈」按钮 → 用户提交 → 消息推送到钉钉群。本文是从零配置到上线测试的完整手册，脱离聊天记录也能照做。

---

## 一、它是怎么工作的

```
用户点「反馈」浮钮
   └─ 弹窗表单（类型 / 描述 / 复现 / 联系）
        └─ POST /api/feedback   （同源，前端已对接）
             └─ handleFeedback(request, env)      ← 唯一逻辑，见下「单一事实来源」
                  ├─ console.log 落一条日志（永不丢）
                  └─ 转发到你的群机器人 webhook → 钉钉群收到消息
```

**为什么端点接了两处**：本项目部署在两家，且都**不是** Cloudflare Pages，所以不能用 CF Pages 的 `functions/` 自动路由，需分别接：

| 线路 | 面向 | 端点承载方式 |
|---|---|---|
| **Cloudflare Workers 静态托管** | 海外 | `site/worker.js`（wrangler 的 `main`）接 `/api/feedback`，其余请求回退静态资源 / SPA |
| **EdgeOne Pages** | 大陆（学生主力） | 构建期把逻辑内联进 `dist/edge-functions/[[default]].js` 的 catch-all，按路径分支 |

两条线路**共用同一份处理逻辑**，行为一致。

### 单一事实来源与文件清单

| 文件 | 作用 |
|---|---|
| `site/functions/_feedback-core.js` | **核心逻辑**（`handleFeedback` + 钉钉加签 + 各家 body 格式）。改逻辑只改这里。 |
| `site/worker.js` | CF Worker 入口：路由 `/api/feedback` → `handleFeedback`，其余 → `env.ASSETS`。 |
| `site/scripts/postbuild.mjs` | 构建期把 `_feedback-core.js` 内联进 EdgeOne 的 catch-all 边缘函数（剥 `export` + 加 `/api/feedback` 分支，整段 `try` 兜底，异常回落 SPA）。 |
| `site/functions/api/feedback.js` | CF **Pages** 薄封装（仅当日后改用 CF Pages 部署才生效，当前用不到）。 |
| `site/wrangler.jsonc` | CF 配置：`"main":"worker.js"` + `assets.binding:"ASSETS"`。 |
| `site/src/components/feedback/FeedbackWidget.tsx` | 前端浮钮 + 弹窗，POST 到 `/api/feedback`；**后端没通时自动降级为"复制反馈内容"**，数据不丢。 |

## 二、第 1 步 · 创建钉钉自定义机器人

1. 在**目标钉钉群** →「群设置」→「智能群助手」→「添加机器人」→ 选 **自定义（通过 Webhook 接入自定义服务）**。
2. 给机器人起名（如「DP 图谱反馈」），可换头像。
3. **安全设置**（钉钉要求至少一项）——二选一：

   | 模式 | 怎么设 | 要不要密钥 | 说明 |
   |---|---|---|---|
   | **加签**（推荐，更安全） | 勾选「加签」→ 复制生成的密钥（`SEC` 开头） | **要**，填进 `FEEDBACK_WEBHOOK_SECRET` | 每条消息带 HMAC-SHA256 签名，代码已实现 |
   | **自定义关键词**（最省事） | 填关键词 `反馈` 或 `DP 图谱` | 不要 | 机器人消息开头就含「🐞 DP 图谱 · 新反馈」，能命中关键词 |

   > 也可用「IP 白名单」，但 Serverless 出口 IP 不固定，不建议。
4. 完成后复制 **Webhook 地址**，形如：
   ```
   https://oapi.dingtalk.com/robot/send?access_token=xxxxxxxxxxxxxxxx
   ```

## 三、第 2 步 · 配置环境变量（CF 和 EdgeOne 各设一遍）

三个变量：

| 变量名 | 值 | 敏感 | 备注 |
|---|---|---|---|
| `FEEDBACK_WEBHOOK_URL` | 第 1 步的 Webhook 地址 | 是 | 含 access_token，务必设为加密 / Secret |
| `FEEDBACK_WEBHOOK_KIND` | `dingtalk` | 否 | **必须设**；不设会按默认 `wecom`（企业微信）格式发，钉钉收不到 |
| `FEEDBACK_WEBHOOK_SECRET` | 加签密钥（`SEC…`） | 是 | **仅加签模式**需要；关键词模式留空 |

### Cloudflare Workers

- **仪表盘**：Cloudflare → Workers & Pages → `dpmaster` → **Settings → Variables and Secrets**
  - `FEEDBACK_WEBHOOK_URL`、`FEEDBACK_WEBHOOK_SECRET` → 类型选 **Secret**（加密）
  - `FEEDBACK_WEBHOOK_KIND` → 类型选 **Text**，值 `dingtalk`
- **或命令行**（在 `site/` 目录）：
  ```bash
  npx wrangler secret put FEEDBACK_WEBHOOK_URL      # 回车后粘贴 webhook 地址
  npx wrangler secret put FEEDBACK_WEBHOOK_SECRET   # 加签密钥；关键词模式跳过这条
  # KIND 非敏感，可在仪表盘加，或写进 wrangler.jsonc 的 "vars": { "FEEDBACK_WEBHOOK_KIND": "dingtalk" }
  ```

### EdgeOne Pages

- 控制台 → Pages → `dpmaster` → **项目设置 → 环境变量** → 新建，作用于**生产环境**，把上表三个都加上。
- 改完环境变量后需**重新部署**才生效。

## 四、第 3 步 · 部署

在 `site/` 目录：

```bash
npm run deploy      # = npm run build && npm run deploy:cf && npm run deploy:eo
```

拆开就是：

| 命令 | 作用 |
|---|---|
| `npm run build` | `tsc -b && vite build`，随后 `postbuild` 生成 EdgeOne 兜底 + 反馈端点 |
| `npm run deploy:cf` | `wrangler deploy`（CF Workers） |
| `npm run deploy:eo` | `edgeone pages deploy ./dist -n dpmaster -e production` |

> ⚠️ **首次 deploy 的变化**：`wrangler.jsonc` 现在有了 `"main":"worker.js"`，CF 项目会从「纯静态」变成「静态 + 极薄 Worker」。**URL 不变**，静态文件仍由资源绑定直出（不进 Worker），只有 `/api/feedback` 进 Worker。深链 / 刷新的 SPA 回退照常（走 `env.ASSETS` 的 `not_found_handling`）。

## 五、第 4 步 · 测试与排查

1. 打开线上站点，点右下角 **反馈**，填一条描述（≥4 字），提交。
2. 钉钉群应**秒收**：

   ```
   🐞 DP 图谱 · 新反馈
   类型：内容有误
   页面：D 矩阵 DP · 网格 / 矩阵上的 DP（/part/d/grid）
   描述：xxxxx
   复现/期望：xxxxx
   联系：（可选）
   环境：1280×800 · Mozilla/5.0 …
   时间：2026-07-06T…
   ```

3. **收不到怎么查**：
   - **CF**：Workers & Pages → `dpmaster` → **Logs**（实时日志），搜 `[feedback]`。
   - **EdgeOne**：控制台 → 函数日志。
   - 日志里 `[feedback] …` 一定有（内容不丢）；若还有 `[feedback] webhook errcode: …` 说明发出去但钉钉拒了，对照下表。

### 钉钉常见 errcode

| errcode | 含义 | 解决 |
|---|---|---|
| `310000` | 签名/关键词校验不通过 | 加签模式：核对 `FEEDBACK_WEBHOOK_SECRET`；关键词模式：确认关键词是 `反馈` 或 `DP 图谱` |
| `130101` | 发送太频繁（限 20 条/分钟） | 教学站正常达不到；短时压测才会触发 |
| `300001` | token 无效 | 核对 `FEEDBACK_WEBHOOK_URL` 是否完整（含 `?access_token=`） |

### 排查速查

| 现象 | 可能原因 |
|---|---|
| 前端提示「已复制，请粘贴到群/邮件」 | 后端没通（端点没部署 / 环境变量没配 / dev 本地没有端点）——这是**兜底**，正常 |
| 日志有 `[feedback]` 但钉钉没消息 | webhook 转发失败，看 `errcode` / `webhook non-2xx` |
| 日志完全没有 `[feedback]` | 请求没到端点：CF 确认已 `wrangler deploy`；EdgeOne 确认边缘函数已随 `./dist` 部署 |
| 钉钉收到但格式乱 | `FEEDBACK_WEBHOOK_KIND` 没设成 `dingtalk`（默认发的是企业微信格式） |

> **EdgeOne 提示**：EdgeOne 边缘函数接 `/api/feedback` 这一路是官方未明文记载的用法，已包 `try` 兜底（异常自动回落 SPA，不影响深链）。**建议上线后在 EdgeOne 线路亲测一次**。CF 那条是标准用法，稳。

## 六、换成别的机器人（可选）

`_feedback-core.js` 同时支持多家，只需改 `FEEDBACK_WEBHOOK_KIND`（body 格式会自动切换）：

| `FEEDBACK_WEBHOOK_KIND` | 平台 | body 格式 | 加签 |
|---|---|---|---|
| `dingtalk` | 钉钉 | `{msgtype:'text', text:{content}}` | 支持（`FEEDBACK_WEBHOOK_SECRET`） |
| `wecom`（默认） | 企业微信 | 同上 | 无需 |
| `feishu` | 飞书 | `{msg_type:'text', content:{text}}` | 未实现（如需另加） |
| `discord` | Discord | `{content}` | 无需 |
| `slack` | Slack | `{text}` | 无需 |

只有钉钉这一家用到 `FEEDBACK_WEBHOOK_SECRET`（加签）。

## 七、隐私与限制

- **匿名**：表单不采集账号 / IP；「联系方式」是**可选**字段，学生愿留才留。
- **内容截断**：描述最长约 1500 字，整包超限返回 413；防止刷爆。
- **不丢数据**：即使 webhook 挂了，反馈内容也会 `console.log` 进函数日志，可事后翻查。
- **频率**：钉钉机器人 20 条/分钟上限，教学站规模远达不到。

## 八、加签算法（备查）

钉钉加签由 `dingtalkSignedUrl()` 实现（`_feedback-core.js`），等价于：

```
timestamp = 当前毫秒
sign = urlEncode( base64( HmacSHA256(secret, `${timestamp}\n${secret}`) ) )
最终 URL = webhook地址 + `&timestamp=${timestamp}&sign=${sign}`
```

用 Web Crypto（`crypto.subtle`）实现，CF Worker / EdgeOne 边缘函数 / CF Pages 三家运行时都原生支持。timestamp 与服务器时间偏差需 <1 小时，Serverless 天然满足。
